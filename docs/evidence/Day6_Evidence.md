## 2. 注册登录功能设计说明

本项目使用 JSON Server 作为 Mock API，因此注册和登录流程都在前端完成。

注册流程：

1. 用户在注册页填写用户名、密码、姓名、学院和年级；
2. 前端先校验必填项和密码长度；
3. 调用 `getUsers()` 查询已有用户；
4. 如果用户名已存在，显示错误提示；
5. 如果用户名不存在，调用 `createUser()` 向 `/users` 写入新用户；
6. 注册成功后跳转到登录页。

登录流程：

1. 用户在登录页输入用户名和密码；
2. 调用 `getUsers()` 获取用户列表；
3. 在前端匹配用户名和密码；
4. 匹配成功后保存到 `userStore.currentUser`；
5. 同时写入 `localStorage`；
6. 跳转到个人中心。

状态恢复流程：

1. 页面刷新或重新打开应用；
2. `App.vue` 调用 `userStore.restoreLogin()`；
3. 从 `localStorage` 中读取 `campus-market-current-user`；
4. 读取成功后恢复 `currentUser` 和 `isLoggedIn`。

退出登录流程：

1. 点击顶部“退出”按钮；
2. 清空 `userStore.currentUser`；
3. 将 `isLoggedIn` 改为 `false`；
4. 删除 `localStorage` 中保存的当前用户；
5. 跳转回首页。

## 8. 遇到的问题与解决方法

遇到的第一个问题是原来的 `userStore` 默认就是已登录用户，如果直接增加登录页，会导致“未登录状态”无法测试。解决方法是把 `currentUser` 改成 `User | null`，默认 `isLoggedIn` 为 `false`，登录成功后再写入用户数据。

遇到的第二个问题是 `currentUser` 可能为空，个人中心和顶部 Header 如果继续直接读取 `currentUser.name` 会报错。解决方法是 getter 中使用安全默认值，同时在个人中心增加未登录分支。

遇到的第三个问题是刷新页面后 Pinia 状态会丢失。解决方法是在登录成功时写入 `localStorage`，并在 `App.vue` 中调用 `restoreLogin()` 恢复当前用户。

遇到的第四个问题是命令行环境中 `node` 不在 PATH 中，直接运行 `pnpm` 会失败。解决方法是使用 Codex 工作区提供的 Node 和 pnpm 路径执行类型检查和构建。

## 10. 今日反思

注册登录功能让项目从“静态模拟用户”变成了更接近真实应用的流程。用户能注册、登录、刷新后保持状态，也能退出登录，这些细节会影响整个系统是否可信。一个项目不能只关注功能能不能跑，还要关注用户当前处于什么状态、操作失败时有没有反馈、刷新后数据是否还能延续。只有把这些体验补齐，校园轻集市才更像一个完整的小型前端应用。
