# Day4 Evidence - 发布表单与数据新增

## 1. 今日完成内容

今天主要完成了“校园轻集市”的发布表单功能。发布页支持四类信息发布：二手交易、失物招领、拼单搭子、跑腿委托。页面可以根据选择的发布类型动态显示不同字段，并对标题、地点、描述、价格、人数、酬劳等内容进行基础校验。

同时补充了四个业务 API 的新增数据方法，分别是 `createTrade()`、`createLostFound()`、`createGroupBuy()` 和 `createErrand()`。表单提交成功后，数据会通过 POST 请求写入 JSON Server 对应集合，并跳转到对应列表页查看新增信息。

## 2. 发布类型与字段设计

| 发布类型 | 对应数据集合 | 关键字段 | 设计理由 |
|---|---|---|---|
| 二手交易 | trades | title、category、price、condition、location、description | 需要展示商品基本信息和交易条件 |
| 失物招领 | lostFounds | title、type、itemName、location、eventTime、description | 需要区分寻物和招领，并描述物品信息 |
| 拼单搭子 | groupBuys | title、type、targetCount、deadline、location、description | 需要说明人数目标和截止时间 |
| 跑腿委托 | errands | title、taskType、reward、from、to、deadline、description | 需要说明任务内容、地点和酬劳 |

## 3. 表单校验规则

通用字段设置了基础必填校验：

- 标题不能为空；
- 地点不能为空；
- 描述不能为空。

不同发布类型还设置了各自的业务校验：

- 二手交易：分类不能为空，成色不能为空，价格必须大于 0；
- 失物招领：物品名称不能为空，时间不能为空；
- 拼单搭子：类型不能为空，截止时间不能为空，目标人数不能少于 2；
- 跑腿委托：任务类型不能为空，取件地点不能为空，送达地点不能为空，截止时间不能为空，酬劳必须大于 0。

如果校验不通过，页面会在对应字段下方显示错误信息，并弹出提示，提醒用户先补全必填内容。

## 7. 遇到的问题与解决方法

遇到的一个真实问题是字段名不一致。提示词中跑腿委托字段写的是 `from` 和 `to`，但项目已有列表页读取的是 `pickupLocation` 和 `deliveryLocation`。如果直接把 `from`、`to` 提交到 `errands` 集合，跑腿列表页就无法正确显示取件地点和送达地点。

解决方法是在发布页提交跑腿委托时进行字段转换：表单内部仍然使用 `from`、`to` 方便理解，发送 POST 请求时改成 `pickupLocation: form.from` 和 `deliveryLocation: form.to`。这样既保留了表单语义，也保证了新增数据能被列表页正常渲染。

另一个问题是当前终端环境中直接运行 `npm` 或 `pnpm` 时可能找不到 Node。解决方法是临时把 Codex 内置 Node 和 pnpm 路径加入当前命令的 PATH，再执行类型检查和构建。

## 8. 今日反思

发布表单、表单校验和数据新增是完整 Web 应用中非常关键的一环。列表页只能展示已有数据，而发布表单让用户真正参与到系统中，能够主动创建内容。表单校验可以减少错误数据进入系统，POST 提交则把前端页面和后端数据连接起来。完成这部分后，项目不再只是静态展示页面，而是具备了基本的数据新增闭环。
